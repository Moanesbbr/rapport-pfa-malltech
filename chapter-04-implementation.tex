\section{Backend Implementation}

\subsection{Application Bootstrap}

The NestJS application is configured at startup with several cross-cutting concerns,
including Helmet for HTTP security headers, CORS restricted to known frontend origins,
global ValidationPipe for DTO validation, express-session stored in MongoDB, and a
custom Socket.IO adapter for WebSocket support.

\subsection{Authentication Module}

The authentication module implements two strategies running in parallel:

\textbf{JWT Strategy} (registered users and admins): \texttt{POST /auth/login}
validates credentials and returns a signed JWT. The \texttt{JwtStrategy} extracts
the Bearer token from the \texttt{Authorization} header on every protected request.
Admin routes use a separate \texttt{AdminAuthGuard} that additionally checks the
\texttt{role} field.

\textbf{Session Strategy} (guest visitors): On the first request, express-session
creates a session document in MongoDB and sets a cookie. The \texttt{SessionGuard}
reads the session ID from the cookie to associate basket operations with a guest.
When a guest registers or logs in, their session basket is merged into their account
basket.

\subsection{Product Catalog Module}

Key implementation details:

\begin{itemize}
    \item \textbf{Slug generation:} Product and category slugs are auto-generated from names, ensuring SEO-friendly URLs.
    \item \textbf{Image handling:} Uploads are processed through Sharp (resize to max 1200px width, WebP conversion, quality 80) and stored on disk.
    \item \textbf{Atomic stock management:} Stock is decremented using MongoDB's \texttt{findOneAndUpdate} with \texttt{\$inc} and a \texttt{quantity >= requested} filter, preventing overselling without distributed transactions.
    \item \textbf{Barcode lookup:} The \texttt{BarcodeLookuService} calls the GO\_UPC API to pre-fill the admin product creation form from an EAN/UPC scan.
    \item \textbf{3D Model viewer:} GLB model files are stored and rendered client-side using a WebGL-based component.
    \item \textbf{Search:} MongoDB's \texttt{\$text} index with text score sort for relevance ranking.
\end{itemize}

\subsection{Commerce Module}

\subsubsection{Order Placement Sequence}

Order placement follows this transactional sequence:

\begin{enumerate}
    \item Load the current basket.
    \item For each item, verify stock availability (\texttt{stock.quantity >= requested}).
    \item Compute line totals and apply any active promotion discount.
    \item Look up the shipping rate for the selected governorate.
    \item Validate and apply a coupon code (if provided), recording usage.
    \item Persist the \texttt{Order} document with status \texttt{pending}.
    \item Atomically decrement stock for each ordered item.
    \item Clear the basket.
    \item Fire-and-forget: send order confirmation email via \texttt{MailService}.
\end{enumerate}

\subsubsection{Invoice and PDF Generation}

When an order is confirmed, an invoice document is created with a sequential number
(counter pattern). PDFKit generates a formatted PDF including the store logo, customer
billing address, line items, subtotal, shipping fee, coupon discount, TVA, and grand
total. The same pattern applies to credit notes.

\subsection{Support Module (WebSocket + Discord Bot)}

The support gateway is a Socket.IO server co-located in the support module.
Customer messages received over WebSocket are saved to the database, forwarded to
all members of the ticket room, and simultaneously sent to a Discord channel via
Discord.js. Replies from Discord agents are forwarded back to the WebSocket room,
completing the bidirectional loop.

\section{Notable Implementation Decisions}

\subsection{Session + JWT Coexistence}
Running both express-session and JWT Passport on the same NestJS app required
careful middleware ordering. Sessions are initialised before Passport, and guards
are applied at the route level to avoid session middleware overhead on API routes
that do not need it.

\subsection{Atomic Stock Decrement}
Stock decrement uses MongoDB's \texttt{findOneAndUpdate} with \texttt{\$inc} and a
\texttt{quantity >= requestedQty} filter. If the filter matches zero documents
(insufficient stock), the operation fails and the order is rejected --- preventing
overselling without a distributed transaction.

\subsection{PDF Generation Without External Services}
Invoice and credit-note PDFs are generated entirely on the backend using PDFKit,
with no dependency on external PDF services. This keeps the system self-contained
and avoids per-document costs.

\subsection{Multi-Library Barcode Scanning}
The admin barcode scanner uses three libraries in cascade because no single library
had reliable support across all target devices and browsers. The cascade tries each
library in order until a successful decode is returned.
